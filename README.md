# not_redis

A Redis-compatible in-memory data structure library for Rust. Provides Redis-like APIs without networking overhead, external dependencies, or operational complexity.

> **Note**: This project was vibe-coded using [OpenCode](https://opencode.ai) and [MiniMax 2.1 Free](https://minimax.io). Code was generated by AI and may contain bugs or edge cases. Review thoroughly before production use.

## Features

- **Strings**: GET, SET, DEL, EXISTS, EXPIRE, TTL, PERSIST
- **Hashes**: HSET, HGET, HGETALL, HDEL
- **Lists**: LPUSH, RPUSH, LLEN
- **Sets**: SADD, SMEMBERS
- **Streams**: XADD, XLEN, XTRIM, XDEL, XRANGE, XREVRANGE
- **Utilities**: PING, ECHO, DBSIZE, FLUSHDB
- **Key expiration** with background sweeper
- **Thread-safe** concurrent access via DashMap
- **Zero-config** -- runs in-process, no server needed

## Installation

```toml
[dependencies]
not_redis = "0.5"
```

Requires a [Tokio](https://tokio.rs) async runtime.

## Quick start

```rust
use not_redis::Client;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let mut client = Client::new();
    client.start().await; // starts background expiration sweeper

    // Strings
    client.set("user:1:name", "Alice").await?;
    let name: String = client.get("user:1:name").await?;
    println!("Name: {}", name);

    // Hashes
    client.hset("user:1", "email", "alice@example.com").await?;
    let email: String = client.hget("user:1", "email").await?;
    println!("Email: {}", email);

    // Lists
    client.lpush("queue", "task-a").await?;
    client.rpush("queue", "task-b").await?;
    let len: i64 = client.llen("queue").await?;
    println!("Queue length: {}", len);

    // Sets
    client.sadd("tags", "rust").await?;
    client.sadd("tags", "cache").await?;
    let tags: Vec<String> = client.smembers("tags").await?;
    println!("Tags: {:?}", tags);

    // Streams
    let id = client.xadd("events", None, vec![("action", "login")]).await?;
    println!("Stream entry: {}", id);

    // Expiration
    client.set("session", "abc123").await?;
    client.expire("session", 300).await?;
    let ttl: i64 = client.ttl("session").await?;
    println!("TTL: {}s", ttl);

    Ok(())
}
```

## Thread safety

not_redis uses `DashMap` for lock-free concurrent access. Share a `Client` via `Arc`:

```rust
use std::sync::Arc;
use tokio::sync::Mutex;
use not_redis::Client;

let client = Arc::new(Mutex::new(Client::new()));
let c = client.clone();

tokio::spawn(async move {
    c.lock().await.set("key", "value").await.unwrap();
});
```

## Error handling

```rust
use not_redis::{Client, RedisError};

match client.get::<_, String>("missing").await {
    Ok(val) => println!("Found: {}", val),
    Err(RedisError::NoSuchKey(k)) => println!("Key not found: {}", k),
    Err(RedisError::WrongType) => println!("Wrong data type for key"),
    Err(RedisError::ParseError) => println!("Could not parse value"),
    Err(e) => println!("Error: {:?}", e),
}
```

## Why not_redis?

| | Redis | not_redis |
|---|---|---|
| Setup | Requires a running server | No setup -- embeds in your binary |
| Network | TCP/IP round-trips | In-process, zero overhead |
| Deployment | Additional service to manage | Ships as part of your app |
| Latency | Microseconds (network) | Nanoseconds (function call) |

## Benchmarks

Measured on Apple Silicon (M1). Redis ran in Docker on the same machine.

| Operation | not_redis | Redis (redis-rs) | Speedup |
|-----------|-----------|-------------------|---------|
| SET | 360k ops/s | 1,300 ops/s | ~277x |
| GET | 356k ops/s | 360 ops/s | ~989x |
| HSET | 348k ops/s | 1,250 ops/s | ~278x |
| HGET | 319k ops/s | 1,400 ops/s | ~228x |
| LPUSH | 358k ops/s | -- | -- |
| RPUSH | 362k ops/s | -- | -- |
| SADD | 351k ops/s | -- | -- |

Batch throughput reaches 4.6--5.2M ops/s for writes and 3.1--3.3M ops/s for reads.

Run benchmarks yourself:

```bash
cargo bench                      # all benchmarks
cargo bench --bench benchmarks   # core operations
cargo bench --bench redis_baseline  # comparison with Redis
```

## Limitations

- No persistence (data lost on process exit)
- No networking layer (in-process only)
- No clustering or replication
- No pub/sub
- No Lua scripting
- No transactions / MULTI-EXEC
- Limited command set (growing)
